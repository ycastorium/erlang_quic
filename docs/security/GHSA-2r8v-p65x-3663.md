# Security report: broken TLS verification

- Advisory: GHSA-2r8v-p65x-3663
- Severity: Critical
- Weakness: CWE-295 (Improper Certificate Validation), CWE-297 (Host Mismatch)
- Package: `quic` (Hex)
- Affected: all versions up to and including 1.4.3
- Reported by: benmmurphy

## Summary

The QUIC client did not authenticate the server. The TLS 1.3 handshake
completed even when the server could not prove its identity. HTTP/3 uses the
same QUIC client, so it had the same problem.

## Impact

An active attacker on the network (man in the middle) can impersonate any
server. The client accepts the connection, so the attacker can read and modify
all the traffic. This breaks the confidentiality and the integrity that TLS is
supposed to give.

## Details

Three checks were missing on the client side:

1. The `CertificateVerify` signature was never checked. This signature is the
   proof that the peer owns the certificate private key. Without it, an attacker
   can present any certificate, even one it does not own.
2. The certificate chain was never validated against a trust anchor.
3. The server hostname was never compared with the certificate.

The `verify` option existed but was never used on the client. Setting
`verify => true` did nothing.

## Fix

The client now, when `verify` is enabled:

1. checks the `CertificateVerify` signature,
2. validates the chain against `cacerts` (or the operating system trust store),
3. checks the hostname against `server_name`.

The client `verify` now defaults to `true`. To connect to a self signed server
(for example in tests), set `verify => false`. To use your own trust anchors,
pass `cacerts => [DerCert, ...]`.

```erlang
{ok, Conn} = quic:connect("example.com", 4433,
                          #{cacerts => [CaDer], server_name => <<"example.com">>},
                          self()).
```

Handshakes that use a PSK (session resumption) are not affected, because no
certificate is sent and the peer is authenticated by the PSK binder.

## Workaround before the fix

There was no real workaround. `verify => true` had no effect, and checking the
certificate after the connection does not help: without the signature check, the
peer is never proven to own the certificate it presents.
